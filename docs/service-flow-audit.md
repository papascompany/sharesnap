## 0. 총평 — 방향은 맞다. 문제는 "설계와 운영 상태의 불일치"다

**"카톡 링크 → 공유방 사진 업로드" UX 차용이라는 설계 방향 자체는 외부 근거로 유효성이 확인된다.**

- 카톡 단톡방 사진 공유의 불만(화질 압축, 유효기간 만료, 용량)은 실재하며 카카오 스스로 유료 구독(톡서랍)으로 수익화 중 — ShareSnap의 "사라지지 않는 원본 공유방"이라는 가치 제안은 성립한다.
- 결정적 방증: **경쟁사 스냅스가 2026-02-04 '공동 포토북'(링크 초대 → 공동 업로드 → 각자 주문)을 런칭**했다. ShareSnap의 핵심 루프와 정면으로 같다 — 시장이 이 모델을 검증하기 시작했다는 뜻이며, 동시에 속도전이 시작됐다는 뜻이다.
- 참여 퍼널 코드(ux-flows.md의 P0 6종: RPC 미리보기·next 보존·콜백 검증·/join SSR 4분기·자동입장)는 **전부 구현 완료**돼 있고 설계 품질도 좋다.

**그러나 현재 운영 상태에서 이 설계는 가동되지 않는다.** 감사의 최상위 결론:

> **카톡으로 초대받은 신규 사용자의 참여 전환율은 현재 구조적으로 0%다.**
> 초대 링크는 카카오톡 인앱 브라우저에서 열리는데, 그 안에서 실동작하는 로그인 수단이 0개이기 때문이다.

이것은 설계 오류가 아니라 "설계의 대전제(카카오 1탭 로그인)를 끈 채 라이브 중"인 운영 상태 문제이며, 해결의 대부분은 코드가 아니라 **반나절의 콘솔 작업 + 소량의 가드 코드**다. 아래 §1이 그 전모다.

## 1. 인지하지 못했던 치명적 오류 — P0 (지금 퍼널이 죽어 있는 지점)

### P0-A. 카카오 인앱 브라우저에서 로그인 수단 0개 [CONFIRMED, 교정 반영]

재현 시나리오(현재 프로덕션에서 100% 재현):

1. 방장이 카톡방에 초대 링크 공유 → 수신자가 탭 → **카카오톡 인앱 브라우저**에서 `/join/{code}` 오픈 (여기까지 정상 — 미리보기 SSR·OG 모두 동작)
2. 미리보기의 CTA는 **카카오 로그인 버튼 단 하나**(`RoomPreview.tsx:112-126` — 대체 수단 없음)
3. 버튼 탭 → Supabase Kakao Provider 미설정이므로 `signInWithOAuth`가 **에러 토스트조차 없이 즉시 리다이렉트**되어, Supabase 도메인의 **raw JSON 400 페이지**(`Unsupported provider: provider is not enabled`)라는 데드엔드에 떨어진다 (검증가가 실제 동작 교정: 클라이언트 catch가 실행되지 않음)
4. `/login`으로 우회해도 인앱 감지(`isKakaoInApp`) 시 **매직링크 폼이 숨겨지고**(`LoginPage.tsx:90-93`) "카카오톡으로 3초 만에 시작할 수 있어요" 문구만 남는다
5. 외부 브라우저 열기 유틸(`openExternalBrowser`)은 정의만 있고 어디에도 노출되지 않음

→ 인앱(=카톡 초대의 기본 컨텍스트)에서 **실동작 로그인 수단 0개**. 60대 부모 페르소나는 물론 30대도 통과 불가.

**원인 구조**: "인앱 여부"와 "Provider 가동 여부"라는 서로 독립적인 두 조건의 분기가 서로를 모른 채 조합되어, 어떤 조합에서 로그인 수단이 0개가 되는지 아무도 검사하지 않았다. 토스 결제에는 있는 graceful 비활성 패턴이 더 앞단인 로그인에는 없다.

### P0-B. 유일한 폴백(매직링크)도 크로스 브라우저에서 구조적으로 실패 [CONFIRMED]

- 발송 메일의 링크가 `/auth/callback?code=`(PKCE code exchange)를 타는데, `code_verifier`는 **로그인을 요청한 브라우저에만** 있다. 모바일의 기본 동선(Safari에서 이메일 입력 → Gmail 앱에서 링크 탭 → 다른 브라우저/웹뷰에서 열림)에서 세션 수립이 실패한다.
- 이를 우회하는 `/auth/confirm`(verifyOtp token_hash — 브라우저 무관) 라우트는 **코드로 완성**돼 있으나, 전제인 **Supabase 이메일 템플릿 ConfirmationURL 매핑이 미반영**(`authService.ts:37-39` 자체 주석 + `integration-inventory.md:189` 기록)이라 실발송 메일은 여전히 깨지는 경로를 탄다.
- **추가 발견(검증가 보강, 더 심각)**: 2024-09 이후 Supabase **내장 이메일은 조직 팀 멤버 주소로만 발송**된다. 운영 프로젝트에 커스텀 SMTP가 없다면 — 지금까지의 매직링크 E2E가 전부 운영자 본인(yohan73) 계정이라 통과했을 뿐 — **일반 사용자는 메일 자체를 못 받고 있을 가능성**이 있다. 대시보드 확인이 필요하다(로컬 config는 SMTP 주석 처리 + 2통/시간 상태).
- 모든 로그인 실패는 `/login?error=`로 리다이렉트되지만 **login 페이지가 error 파라미터를 읽지 않아**(`login/page.tsx:9` — next만) 사용자는 아무 설명 없는 로그인 화면만 본다. "랜덤 실패 + 무언(無言)"의 조합.

### P0-C. 결제를 켜는 순간 전자상거래법 요건 미비 상태로 판매 개시 [CONFIRMED]

- `/terms`·`/privacy` 페이지 부재. 로그인 화면의 "동의 간주" 문구는 **링크조차 없다** — 존재하지 않는 문서에 대한 동의 간주는 효력이 없고, 개인정보보호법 제30조(처리방침 공개 의무) 위반 상태다. 이메일·얼굴 사진·배송지·결제를 수집하는 서비스다.
- 청약철회: 주문제작 재화(포토북/인화)는 철회 제한이 **합법이지만 사전 고지가 성립 요건**인데 고지할 문서가 없다. 환불 개시 경로도 0개(토스 취소 API 호출 코드 전무 — 토스 API는 confirm 1개뿐, `paymentServer.ts:261`). 토스 가맹 심사도 약관·환불규정 URL을 요구한다.
- placeholder 가격이 결제 활성화 스위치(토스 키)와 묶여 있음: 키를 넣는 순간 원가 미검증 가격으로 실판매가 시작된다. STATUS.md의 "토스 키만 남음 — 등록하면 즉시 활성" 안내가 오히려 단독 실행을 유도하는 상태(검증가 교정).
- 계획된 Play Store TWA / iOS 심사도 privacy policy URL이 필수 — 이 페이지들 없이는 제출 자체가 불가능.

### P0-D. 삭제한 사진의 공개 썸네일 3종(3600px 인쇄용 포함)이 영구 잔존 [CONFIRMED]

- `thumbnails` 공개 버킷에 **DELETE RLS 정책이 아예 없다**(`007:36-45` — select/insert만). 클라이언트의 삭제 요청은 정책상 통과되지 않는데 `deletePhoto`는 Storage 삭제 실패를 삼킨다(best-effort).
- 결과: 사용자가 사진을 지워도 thumb 400px·medium 1280px·**print 3600px**이 만료 없는 public URL로 전 세계에 무기한 노출. "삭제된 사진"의 원본급 이미지가 URL만 알면 계속 열린다. 개인정보보호법 제21조(파기)·삭제권에 응답할 수 없는 구조이고, 세션 #9~10에서 공들인 "사진 삭제 UX"의 약속이 스토리지 레이어에서 배신당하고 있다.

## 2. 사용자 초대 — 현재 구현 기준의 구체적 운영 시나리오와 살리는 법

### 2.1 지금 코드가 제공하는 초대 플로우 (전 구간 구현 확인됨)

```
[방장]  방 화면 헤더 → 초대 버튼(/rooms/{id}/invite)
        → ①링크 복사 ②OS 공유시트 ③카카오톡 공유(Feed 카드) ④초대 코드 표시
[수신자] 카톡 카드/링크 탭 → /join/{code}
        → 비로그인: SSR 미리보기(방이름·커버·멤버수·사진수) + "카카오로 시작하기"
        → 카카오 1탭 → /auth/callback?next=/join/{code}?auto=1 → 자동 입장
        → /rooms/{id}?welcome=1 → "'{방이름}'에 참여했어요" 토스트
        → 기존 멤버가 재탭하면: 미리보기 생략, 즉시 방으로 (1탭 재입장)
[재초대] 사진 업로드 완료 시트 → "카톡방에 알리기"(새 사진 N장 Feed 카드)
```

설계(ux-flows.md)와 코드가 일치하며, 카카오만 켜면 "신규 3탭 / 재방문 2탭 / 세션 보유 1탭"이 그대로 성립한다.

### 2.2 이 플로우를 실제로 켜는 체크리스트 (반나절, 외부 콘솔 작업)

카카오 로그인의 통념적 허들("이메일 동의항목 = 비즈 앱 = 사업자등록 + 심사")은 **2026-07 현재 사실이 아님이 리서치로 확인됐다**: 개인 개발자는 본인인증 + [개인 개발자 비즈 앱 전환] 버튼으로 당일 자가 전환이 가능하다(2026-07-03 카카오 데브톡 담당자 답변 — 공식 공지문은 미발견이므로 콘솔에서 버튼 실확인 필요). 이메일을 포기하는 경로("Allow users without an email")도 있으나 **권장하지 않는다**(§4.3 계정 이산 참조).

| # | 작업 | 위치 |
|---|------|------|
| 1 | 카카오디벨로퍼스 앱 생성 + 앱 아이콘 등록 | 카카오 콘솔 |
| 2 | 본인인증 → [개인 개발자 비즈 앱 전환] (이메일 동의항목용) | 앱 > 비즈니스 |

| 4 | 카카오 로그인 활성화 + Client Secret 발급·활성화 | 제품 설정 |
| 5 | Redirect URI: `https://rtnfltwmnizkjrrgjudk.supabase.co/auth/v1/callback` | 카카오 로그인 |

### 2.3 초대 발신측도 고칠 것 (카카오를 켜도 남는 문제)

- **초대 카드 스펙 미달**: invite 페이지가 커버/사진수/멤버수를 조회하지 않아(`invite/page.tsx:13-38`) 카톡 카드가 앱 아이콘 + 제네릭 문구로 나간다. 아이러니하게 "링크 복사→붙여넣기"의 OG 미리보기가 공식 공유 버튼보다 품질이 높은 역전 상태. → 이미 있는 `get_room_preview` RPC를 호출해 전달만 하면 됨(30분).
- **에러 위생**: 키 미설정 시 카카오 공유 버튼이 `"NEXT_PUBLIC_KAKAO_JS_KEY 환경변수가 없습니다"`를 사용자 토스트로 노출(`KakaoShareButton.tsx:53-54`), "카톡방에 알리기"는 실패를 console.warn으로 삼켜 **재초대 루프가 조용히 0건으로 죽는다**(`PhotoUploader.tsx:118-135`). → `isKakaoShareConfigured()` 게이트로 미설정 시 버튼 숨김 + 링크 복사 승격, 실패 토스트 추가.
- **재발 방지 불변식**: "어떤 (인앱 여부 × Provider 설정) 조합에서도 실동작 로그인 수단 ≥ 1개"를 코드로 강제 — `NEXT_PUBLIC_KAKAO_AUTH_ENABLED` 플래그로 카카오 버튼을 조건부 렌더하고, off면 인앱에서도 매직링크 폼을 노출(템플릿 매핑 선행 조건).

## 3. 세션 유지 — 검증 결론

**정책 레이어는 문제없다. 리스크는 전부 클라이언트 저장소에 있다.**

| 검증 항목 | 결론 |
|---|---|
| 세션 수명 | Supabase 기본값 = refresh token 무만료·1회용 rotation, timebox/inactivity 미설정 → **스토리지가 살아있는 한 세션은 사실상 무기한**. 장기 미접속 복귀도 문제없음 |
| 토큰 갱신 | proxy(`src/proxy.ts`)→`updateSession`이 매 요청 자동 갱신 — 정상 |
| 인앱 웹뷰 저장소 | **공식 보장 없음.** 데브톡에 랜덤 로그아웃 사례 보고(카카오 무응답), 카톡 앱 캐시 삭제에 종속. "인앱 세션은 언제든 사라진다"를 전제해야 함 |
| 컨텍스트 분리 | 인앱 ↔ 외부 브라우저 ↔ PWA 저장소는 완전 분리 — 컨텍스트당 1회 로그인은 설계상 감수(카카오 1탭이면 비용 낮음). PWA 설치본이 유일하게 소실 리스크 없는 컨텍스트인데 **설치 유도(§5.3)가 전면 미구현** |
| 재방문 복귀 | 주 경로(카톡방의 `/join` 링크 재탭)는 **끊기지 않는다** — 기존 멤버 즉시 리다이렉트 + next 자체 처리 완비. 끊기는 것은 보호 경로 직접 딥링크(북마크·복사한 방 URL·주문 링크): **미들웨어가 next를 버려**(`middleware.ts:55-59`) 로그인 후 방 목록에 착지. 페이지 레벨 가드 7곳이 부착하는 next도 미들웨어가 먼저 가로채 전부 사장 → **`url.searchParams.set("next", pathname + search)` 한 줄**로 기존 인프라 전체가 살아남 |
| 체류 중 만료 | AuthGuard가 구현만 있고 마운트 0곳(데드코드) — 채팅 체류 중 세션이 죽으면 화면은 멀쩡한데 전송만 실패. (main) 셸에 마운트 + next 부착으로 해소 |
| **계정 일관성** | **가장 큰 숨은 리스크.** 현재 유저는 전원 이메일(매직링크) identity. 카카오를 "이메일 없이" 켜면 같은 사람이 두 계정으로 갈라져 방·주문 이력이 영구 분리되고 병합 수단이 없다. ADMIN_EMAILS 게이트도 이메일 없는 계정과 충돌. → **account_email 필수 동의 + "Allow users without an email" OFF**가 정답(자동 연결은 카카오 이메일이 기존과 동일할 때 성립 — 보장이 아닌 최선 경로이므로, 로그인 화면에 "이전에 이메일로 로그인했다면 같은 이메일의 카카오 계정이어야 데이터가 유지됩니다" 1줄 고지 + 중기적으로 `linkIdentity()` 기반 계정 연결 버튼) |

**세션 정책 결론**: "세션을 오래 유지하려 애쓰기"보다 **"세션이 죽어도 재인증이 1탭이고, 재인증 후 원위치로 복귀한다"**가 이 서비스의 올바른 세션 전략이다. 그 전제가 ①카카오 로그인 가동 ②미들웨어 next 1줄 ③welcome 직후 PWA/외부 브라우저 유도(비공식 `kakaotalk://web/openExternal` 스킴 + 실패 시 URL 복사 폴백 — 실기기 검증 후 배포)다.

## 5. 운영 정책 공백 — 서비스 정책 최적화 제안

### 5.1 방장 통제권 (링크 유출 대응) — 현재 0개

카톡 링크 유통이 핵심 퍼널인 이상 유출은 일상인데, 유출 시 방장이 할 수 있는 조치가 **문자 그대로 0개**다: share_code 재발급 없음, 방 잠금 없음, 강퇴는 RLS가 이미 허용하는데 UI 없음, 방 삭제/나가기는 서비스 함수만 있고 호출처 0(데드코드). share_code 생성도 Math.random(비암호학적).

**처방(1일)**: `rooms/[id]/settings` 페이지 신설 — ①초대 링크 재발급(share_code update, 방장 RLS 이미 존재 / 생성기 `crypto.getRandomValues` 교체) ②멤버 리스트+강퇴(RLS 완비, UI만) ③deleteRoom/leaveRoom 배선(+storage 정리 service_role 라우트). 2단계: `is_locked` 컬럼 + join RPC 검사. **이미 지어진 자산에 UI만 배선하는 것이라 투자 대비 효과가 가장 크다.**

### 5.2 신고·모더레이션 — 최소 이행선

신고·차단 0건, 방장도 타인 코멘트 삭제 불가, 운영자가 문제 콘텐츠를 인지할 경로 자체가 없음. 폐쇄형 방이라 정보통신망법 임시조치 의무가 당연 적용된다고 단정할 수는 없으나(검증가 교정), **공개 버킷 public URL 구조가 "일반 공개 유통" 해석을 불리하게 만들고**, 피해자·기관의 삭제 요청이 왔을 때 접수·처리 수단이 없으면 그 시점의 의무 이행이 불가능하다.

**처방(1~2일)**: ①PhotoViewer 신고 버튼 + reports 테이블 ②admin 신고 큐 + service_role 삭제(사진+썸네일 동시 파기) ③약관 금지조항·처리 절차 ④신고 이메일 게재. 법이 요구하는 것은 자동 필터링이 아니라 "창구와 절차의 존재·이행"이다.

### 5.3 상한 3종 (악용 시나리오 천장)

방 인원·누적 업로드·방 생성 수 전부 무제한(1회 30장 제한만 — 반복 호출로 무의미) + 공개 썸네일 핫링크 가능 = 오픈채팅 링크 유포 한 번이면 스토리지·egress 비용 폭탄, 지울 UI도 없음.

**처방(각 10줄 미만)**: ①join RPC에 인원 검사(예: 100명 — security definer 내부라 우회 불가) ②업로드 전 방 사진 수 검사(예: 2,000장) ③방 생성 상한(예: 20개). 정상 사용자(여행 1회 수백 장)는 체감하지 않는 값으로 시작 — "상한이 존재한다"는 사실 자체가 방어다.

### 5.4 보존·탈퇴 정책 (처리방침의 입력값이자 iOS 블로커)

- 가치 제안이 "사라지지 않는 사진"인데 보존 기한 정책이 없다 → 무기한이면 비용 단조 증가, 기한을 두면 가치 제안과 충돌 — **의식적 결정 필요**: "마지막 활동 후 N년 보존, 삭제 N일 전 방장 고지"를 확정해 처리방침에 명시(초기엔 cron 없이 정책 문구만으로 충분).
- **회원 탈퇴 기능 0건** — 개보법 동의 철회 대응 불가 + **Apple 심사 5.1.1(v)(인앱 계정 삭제 의무)로 iOS 배포 블로커**. 탈퇴 시: 프로필·멤버십 즉시 파기, 사진은 "함께 삭제" 기본 + "방에 남기기(익명 귀속)" 선택, 결제 이력은 계정과 분리 보존(전상법 5년), 방장 탈퇴 시 위임(room_members UPDATE 정책 신설) 또는 최고참 자동 위임.

### 5.5 "누가 올렸는지"가 어디에도 없다

profiles 테이블·트리거 부재, 채팅 말풍선·사진·코멘트에 작성자 닉네임/아바타 미표시(코멘트는 "나"/"멤버" 고정). **단톡방 UX를 차용한 서비스에서 발화자 식별 불가는 정체성 결손**이다. 카카오 로그인 가동 시 프로필(닉네임·아바타)이 들어오므로, profiles 테이블 + auth 트리거 + 말풍선/코멘트 표시를 카카오 활성화 직후 스프린트로.
